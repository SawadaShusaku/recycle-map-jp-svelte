## Context

現在のポップアップ（`+page.svelte`の`popupCard`）では、施設の基本情報タブに`notes`フィールドの内容をそのまま表示している。`notes`は`f.properties.notes`（GeoJSONのproperties）から取得されており、データベースでは`facilities.notes`および`place_collection_entries.notes`に格納されている。

`sanitizePublicText`関数（`src/lib/public-data-quality.ts`）は、ジオコーディング関連の内部メモを除外する仕組みを持つが、開館時間や施設種別名（「自治体施設」「郵便局」など）の混入に対しては対処していない。これらの文字列はデータ収集パイプライン（private）で誤って`notes`欄に含められた可能性が高い。

## Goals / Non-Goals

**Goals:**
- 現在のデータに含まれる`notes`の実態を調査し、不要な文字列の混入パターンを特定する
- `sanitizePublicText`の除外パターンを拡張し、不要な文字列を公開APIから除外する
- 必要に応じて、フロントエンドでの表示制御またはデータ移行を行う

**Non-Goals:**
- privateデータパイプライン自体の修正（本リポジトリ外のため、方針の提示に留める）
- `hours`（営業時間）フィールドのデータ品質改善（別課題）
- 全データの手動レビュー（自動検出に限る）

## Decisions

### 1. 調査アプローチ：SQLベースのサンプリング調査
- 理由：production D1またはlocal D1に対してSQLを発行し、`notes`の内容を集計・サンプリングすることで、実際の混入パターンを把握する。推測ではなくデータに基づいて対処する。
- 代替案：コードベース内のテストデータのみ確認 → 棄却。実データのパターンが反映されない。

### 2. 修正レベル：APIサニタイズ層（`sanitizePublicText`）を拡張
- 理由：データベースの既存レコードを直接修正する前に、APIレスポンス層で不要データを除外することで、影響範囲を限定し、安全に検証できる。
- 代替案：DBマイグレーションで一括削除 → 棄却。誤削除リスクがあり、ロールバックが困難。

### 3. パターン定義：正規表現ベースの柔軟なマッチング
- 理由：「開館時間」や「郵便局」のような混入は、データソースによって表記揺れがある可能性がある。正規表現で「開館」「閉館」「営業時間」などを広くカバーする。
- 代替案：完全一致リスト → 棄却。表記揺れに対応できない。

## Risks / Trade-offs

- **[Risk] `sanitizePublicText`のパターンが有用なNoteまで除外してしまう** → Mitigation: パターンは文字列の先頭または全体マッチに限定し、「入口横」のような具体的な位置情報は誤って除外しないようにする。テストケースを充実させる。
- **[Risk] データパイプラーン側の修正が行われないため、同じ問題が再発する** → Mitigation: 調査結果をドキュメント化し、private pipeline側にフィードバックする仕組みを提案に含める。
- **[Risk] `notes`に有用な情報と混入情報が混在している** → Mitigation: 混在パターンが見つかった場合、部分的なクリーニング（正規表現で部分削除）ではなく、公開表示からの除外を優先する。

## Migration Plan

1. **調査フェーズ**：local D1またはproduction D1から`notes`のサンプルを抽出し、混入パターンを特定
2. **設計フェーズ**：特定したパターンをもとに、`INTERNAL_NOTE_PATTERNS`への追加を設計
3. **実装フェーズ**：`public-data-quality.ts`を更新し、テストを追加
4. **検証フェーズ**：`npm run smoke`および手動でのポップアップ表示確認
5. **必要に応じて**：DBレベルでの一括クリーニング（別途スクリプト化を検討）
